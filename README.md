# TikTok 简化直播间 Android 技术文档

一个高性能的 Android 直播应用，采用 MVVM 架构，支持 DASH 直播流播放、实时评论、在线人数统计等功能。通过多种性能优化技术，实现了流畅的直播观看体验。

## 项目简介

本项目是一个仿 TikTok 直播间的 Android 应用，主要功能包括：

- **DASH 直播流播放**：使用 Native组件播放，支持流畅的直播流播放
- **实时评论**：支持发送和接收实时评论，评论列表自动滚动
- **实时人数**：通过 WebSocket 消息通道，进入直播间时，直播间人数+1
- **性能优化**：预加载、视图池、TTL 缓存等多种优化技术
- **插件化架构**：支持点赞、礼物、聊天等插件功能，实现即插即用

## 设计思路

本项目以直播类应用"高实时性、高并发性、高资源消耗"的业务特性为基础，从用户真实使用场景出发，围绕"进入应用—选择直播间—观看互动—退出回流"这一完整使用路径中的关键体验痛点进行系统性设计。

**核心体验目标**：

- **快速进入**：减少应用启动和进入直播间的等待时间
- **流畅观看**：保证视频播放的稳定性和连贯性
- **即时互动**：确保评论、在线人数等实时数据的及时响应
- **快速回流**：实现用户短时间内重新进入直播间的低延迟体验

**核心关注点**：

- 减少等待时间
- 降低卡顿现象
- 提升连贯性和沉浸感

**性能优化策略**：

针对用户最容易感知到卡顿和等待的关键场景进行优化：

- 应用启动
- 进入直播间首帧加载
- 评论高频刷新
- 用户短时间内频繁进出直播间

从四个维度进行针对性优化：

- **启动链路**：通过预加载机制缩短首次等待时间
- **播放链路**：优化播放器配置和缓冲策略，保证播放流畅
- **交互链路**：通过视图池复用与数据增量刷新降低界面刷新开销，结合防抖策略减少无效UI更新
- **回流链路**：通过TTL缓存机制实现低延迟回流

**最终目标**： 从整体上保证用户在高频操作和长时间观看过程中的流畅性与稳定性，最终实现"快进、稳播、顺畅互动、无感回流"的观看体验。

## 核心功能

### 1. 启动流程

- **SplashActivity**：启动页，显示应用 Logo，后台预加载首页组件

- 在主页面预加载 WebSocket确保启动后快速进入直播间
- 在直播间点击右上角的×号退出到主界面



### 2. 页面详情

- SplashActivity：启动页，显示tiktoklogo
- MainActivice：主页面，仿照抖音直播页面复刻

- **LiveRoomActivity**：直播间主界面
  - 左上角：主播头像、主播名称、关注数量
  - 右上角：在线观众数量
  - 下方区域（公屏聊天区）：用户头像、名字、聊天内容
  - 底部区域：评论框，发送按钮，点赞键

### 3. 性能优化

- **预加载**：在主页时预加载WebSocket，预渲染 Activity 布局，减少首次渲染时间
- **直播间缓存**：正常退出后30秒内重新进入可以快速恢复完整状态



## 技术栈

### 架构

- **MVVM架构**：职责分离，解耦
- **Repository**：统一数据访问接口
- **插件化架构**：符合开闭原则，支持多功能扩展

### 核心技术

- **Native播放器**：播放dash流组件
- **OkHttp**：网络请求库
- **Gson**：JSON 解析
- **Glide**：网络图片加载和缓存
- **WebSocket**：实时通信（在线人数、评论）

### Android 组件

- **Lifecycle & LiveData**：生命周期感知，响应式数据更新
- **ViewModel**：管理 UI 相关数据，独立于 Activity 生命周期
- **RecyclerView**：列表展示，支持 DiffUtil 优化
- **ConstraintLayout**：灵活的布局系统

## 核心流程

启动页（2 秒）→ 主页面（提供入口 + 预加载）→ 直播间页面（视频播放 + 互动 + 缓存管理）

### 启动页与主页面流程

```
1. SplashActivity.onCreate()
   ├── 初始化布局
   ├── 通过Handler设置 2 秒延迟，延迟结束后跳转到主页面
   ├── 调用finish()销毁当前启动页

2. MainActivity.onCreate()
   ├── 初始化布局
   ├── 设置按钮点击事件,点击后跳转直播间
   ├── 预加载资源
   │   ├── 预加载 WebSocket
   │   ├── 预加载网络连接资源

3. 跳转到 RoomListActivity
```



### 直播间流程

```
1. 用户点击进入直播间按钮
   └── 启动 LiveRoomActivity
   
2. 加载数据
   ├── 获取主播信息
   ├── 获取评论列表
   └── 连接 WebSocket（增加在线人数）
   
3.初始化
   ├── 初始化控件：绑定主播信息（头像、名称、在线人数）、评论列表（RecyclerView）、评论输入框、点赞按钮、视频播放器等。
   ├── 初始化辅助组件：ViewModel（管理数据与 UI 交互）、LikePlugin（处理点赞逻辑）、LiveRoomCacheManager（管理缓存）
   ├── 检查缓存
   │   ├── 若存在有效缓存，从缓存恢复状态（主播信息、评论列表、视频播放进度、点赞数等）
   │   └── 若无有效缓存，通过ViewModel注册数据观察者（监听评论、主播信息、在线人数等变化），加载初始数据，建立WebSocket连接，并初始化视频播放。

4. 数据交互与 UI 更新
   └── 数据加载：通过ViewModel加载主播信息、评论列表等数据，加载完成后自动更新 UI。
   └── 实时互动
   │   ├── 评论功能：用户输入评论后，点击提交按钮，通过ViewModel处理提交逻辑，成功后刷新评论列表并滚动到最新评论。
   │   ├── 点赞功能：点击点赞按钮，通过LikePlugin处理点赞逻辑，实时更新点赞数。
   │   └── 通过ViewModel监听实时在线人数变化，更新显示。
   └── 视频播放
   │   └── 使用ExoPlayer加载视频源，支持播放、暂停、进度调整，从缓存恢复时可还原播放进度和状态。   
```



### 退出直播间流程

```
1. 主动触发退出（点击返回按钮）
   ├── 触发入口：直播间页面的返回按钮点击事件。
   ├── 通过LiveRoomCacheManager将直播间当前状态写入缓存
   └── 释放资源 防止内存泄露
```



## 遇到的问题以及解决思路

### 一、开发阶段遇到的问题

#### 问题1：应用使用时间过长，频繁切换页面时出现卡顿

**问题描述**： 当模拟器使用时间过长，切换页面次数过多时，模拟器会愈发卡顿

**问题分析**：

​	切换页面时没有销毁当前页面，占用资源，模拟器内存不够，导致卡顿

**解决方案**：及时释放资源，防止内存泄露

**优化效果**：

- 方法耗时大幅降低，及时使用时间过长也不会过分卡顿



#### 问题2：xgplayer无法播放DASH直播流

**问题描述**： 使用xgplayer播放DASH格式的直播流时，若使用dash播放器一直显示"缓冲中"状态，无法正常播放。使用shark组件时无法解析mpd文件，但用播放器组件播放本地视频以及网络视频时没有问题

**问题分析**：

- `xgplayer-dash`插件无法正常播放DASH直播流，应使用`xgplayer-shaka`插件，但是英文站的官网在线例子js插件路径有误，中文站的版本播放成功，但是卡顿
- 网络问题

**解决过程**：

1. 对于xgplayer播放DASH直播流的问题并没有解决，组件无法导入，我尝试过所有可能的方法，大部分推荐的版本，都无法正常导入插件，若能正常导入插件也无法正常解析mpd文件。经群里讨论，大概率是网络问题
2. 最后解决方案，使用Native组件成功播放

**经验总结**：

- 需要根据实际需求选择可行的技术方案，要考虑兼容行
- 通过demo项目验证技术的可行性，再去对项目进行集成



#### 问题3：在退出直播间再次进入时，websocket重复发送导致在线人数一直在增加

**问题描述**：正常流程进入直播间时通过websocket发送消息，若成功接收则在线人数+1，若失败，由于在线人数可以不保证数据强一致，失败遍重试，但是退出重新进入时直播间人数无限递归的+1

**问题分析**：退出直播间时没有断开当前Activity的Websocket连接，导致再次进入时再次调用，由于是单例模式，导致无限递归发送消息

**解决过程**：退出时清除当前页面，保留对应数据加入缓存，这点在问题一的解决方法是一样的



#### 问题4：在添加直播间缓存后，问题3又出现

**问题描述**：同问题三

**问题分析**：在 initView() 方法中的+1操作会在每次进入直播间时都执行，导致每次都向服务器发送"1"消息，从而使在线人数不断增加。

**解决过程**： 仅在不是从缓存恢复的情况下发送文本消息



#### 问题5：频繁添加评论导致卡顿

**问题描述**： 在通过接口发送评论消息时，需要从接口重新获取所有评论，频繁会导致卡顿

**问题分析**：

- 频繁获取接口以及频繁渲染评论必然导致卡顿

**解决方案**：在提交成功后，直接将数据添加在列表末尾，并且只刷新最后一项

**优化效果**：

- 评论添加时，只渲染最后一项，性能提升



#### 问题6：频繁进入直播间时启动时间过长

**问题描述**： 每次进入直播间时，直播加载较慢，若频繁退出进入，性能和体验都不是很好

**问题分析**：

- 每次进入时都要重新加载组件，必然导致性能差

**解决方案**：

- 预加载WebSocket连接
- 主动退出直播间时 ，缓存直播间相关数据，在30s内，如果重新进入直播间可以快速恢复



#### 问题7：评论数量的显示

**问题描述**： 在实际场景中，评论的数量应控制在一定范围内，由于接口的内容只能存储100条数据，我设置最大显示评论数为100，但是更多的评论数量对性能的影响无法测试
